## Eureka Insights

1. **Naming inversion is deliberate, not accidental.** arifOS uses `noun_verb` for registry grouping. WELL uses `verb_noun` for English readability ("classify substrate" > "substrate classify"). Both are valid if documented.

2. **Alias layer is the only safe migration pattern for MCP tool renames.** MCP clients cache tool lists. Hard cutovers break persistent connections. 13 aliases cost nothing and prevent federation outages.

3. **Coupled readiness is WELL's true product.** Human-only readiness is incomplete. Machine-only reliability is incomplete. MCP-only health is incomplete. The intersection (`human + machine + MCP`) is what arifOS actually needs to judge safely.

4. **WELL's scope is "substrate truth at the body/livelihood level."** arifOS's scope is "system-level abstraction, routing, and verdicts." This division is clean and defensible. WELL must never judge; arifOS must never measure biology.

5. **The packet envelope must separate reflection from anchoring.** `well_get_packet` reads. `well_anchor_evidence` writes. Conflating them breaks audit trails and creates confusion about side effects.

6. **arifOS doesn't need WELL's internals.** arifOS only needs: `coupled_verdict`, `human_ready`, `machine_ready`, `mcp_ready`, `well_score`, `confidence`. Everything else is dashboard/debug detail.

7. **Foundation.md anchors the ontology.** The chemistry → biology → body → livelihood → machine → intelligence → dignity → audit ladder gives WELL a scientific north star. Every tool name maps to a rung on this ladder.
